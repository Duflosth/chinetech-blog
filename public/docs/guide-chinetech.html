Guide simple pour non-technicien

Gérer le blog CHINETECH

Ce guide explique les services, les comptes à créer, comment écrire des articles, et comment demander à Codex de modifier le site sans rentrer dans la technique.

Les services

Voici les services à connaître. Il ne faut pas les apprendre comme une liste technique. Il faut plutôt les voir comme les personnes qui font tourner un petit magazine.

GitHub : l'armoire officielle

Explication générale GitHub est l'endroit où le projet vit réellement. Il garde les fichiers du site, les articles, les images importantes et l'historique des changements. Dans notre métaphore, c'est l'armoire officielle du magazine : si quelque chose est dans GitHub, le projet peut le retrouver.

Contexte de création Les projets web ont besoin d'un endroit commun où plusieurs personnes peuvent travailler sans écraser le travail des autres. GitHub répond à ce besoin : garder le code, garder les versions, et permettre de revenir en arrière si une erreur est faite.

Exemple Si Codex modifie la page d'accueil, GitHub conserve cette modification avec un message. Plus tard, on peut retrouver ce changement et comprendre ce qui a été fait.

Netlify : l'imprimeur et la vitrine

Explication générale Netlify prend ce qui est rangé dans GitHub et le transforme en site visible par le public. Dès qu'un changement est envoyé dans GitHub, Netlify reconstruit le site et le met en ligne.

Contexte de création Avant ce type de service, publier un site demandait souvent de gérer soi-même un serveur. Netlify simplifie cela : on branche le site à GitHub, et la publication devient automatique.

Exemple Quand un article passe de draft: true à draft: false, Netlify reconstruit le site. Après quelques instants, l'article apparaît sur chinetech.netlify.app.

Decap CMS : le bureau de rédaction

Explication générale Decap CMS est la page d'administration accessible sur /admin/. C'est là qu'une personne non technique peut écrire un article avec un formulaire. Elle n'a pas besoin de connaître le code, ni d'ouvrir GitHub.

Contexte de création Un site statique est rapide et simple à héberger, mais écrire directement dans les fichiers peut être intimidant. Un CMS sert à donner une interface de rédaction claire au-dessus de ces fichiers.

Exemple Un rédacteur ouvre /admin/, clique sur "New Article", écrit son texte, sauvegarde, puis le CMS envoie l'article vers GitHub.

Netlify Identity : le badge d'entrée

Explication générale Netlify Identity sert à savoir qui a le droit d'entrer dans le CMS. Une personne invitée reçoit un email, crée son mot de passe, puis peut se connecter à /admin/.

Contexte de création Un espace d'administration ne doit pas être ouvert à tout le monde. Identity apporte une porte d'entrée : seules les personnes invitées peuvent créer ou modifier des articles.

Exemple Pour ajouter un rédacteur, le propriétaire va dans Netlify, ouvre Identity, envoie une invitation email, et le rédacteur choisit son mot de passe.

À retenir Il ne faut pas créer un compte Netlify complet pour chaque rédacteur. Pour écrire des articles, une invitation Identity suffit.

Git Gateway : le coursier discret

Explication générale Git Gateway est le lien entre le CMS et GitHub. Quand un rédacteur clique sur sauvegarder, Git Gateway transporte le changement vers l'armoire GitHub.

Contexte de création Un rédacteur CMS ne doit pas avoir à manipuler GitHub directement. Git Gateway existe pour permettre au CMS d'envoyer des changements dans GitHub au nom d'un utilisateur autorisé.

Exemple Si un article est sauvegardé dans le CMS, Git Gateway crée le changement côté GitHub. Si la sauvegarde échoue, c'est un des premiers réglages à vérifier dans Netlify.

Astro : la machine de mise en page

Explication générale Astro est l'outil qui transforme les fichiers du projet en pages web rapides. Il prend les articles, les styles, les pages et les assemble pour produire le site final.

Contexte de création Les sites modernes ont besoin d'être rapides, organisés et faciles à maintenir. Astro sert à fabriquer des pages statiques propres à partir de composants et de contenus.

Exemple Quand Netlify lance la publication, Astro prend les fichiers Markdown des articles et fabrique les pages visibles dans la rubrique Articles.

Codex : l'assistant pour modifier l'atelier

Explication générale Codex sert quand il faut changer le site lui-même : modifier la page d'accueil, corriger un bug, changer un style, ajouter une page, ajuster le CMS ou produire un document comme celui-ci.

Contexte de création Modifier un site demande souvent de comprendre plusieurs fichiers. Codex sert d'assistant de code : on lui décrit le résultat voulu, il lit le projet, propose les modifications et vérifie que ça fonctionne.

Exemple Si tu veux changer la section newsletter, tu demandes à Codex : "change ce texte, ne casse pas le formulaire, lance le build, puis pousse si tout est bon".

Résumé très court : GitHub garde, Netlify publie, Decap CMS permet d'écrire, Identity ouvre la porte, Git Gateway transporte, Astro assemble, Codex modifie le site quand on lui demande.

Quels comptes créer ?

Il ne faut pas créer tous les comptes pour tout le monde. Chaque personne doit avoir seulement les accès qui correspondent à son rôle.

Le propriétaire du projet

Explication générale Le propriétaire doit avoir un compte GitHub, un compte Netlify et un accès ChatGPT/Codex. C'est la personne qui peut gérer le site, inviter quelqu'un, vérifier les déploiements et demander des changements plus profonds.

Contexte de création C'est l'équivalent du directeur de publication : il n'écrit pas forcément tous les articles, mais il garde les clés principales.

Exemple Si un rédacteur perd son accès, le propriétaire peut renvoyer une invitation depuis Netlify et vérifier ensuite que le CMS fonctionne.

Le rédacteur non technique

Explication générale Un rédacteur qui écrit seulement des articles n'a pas besoin d'un compte GitHub. Il a besoin d'une invitation Netlify Identity pour entrer dans le CMS sur /admin/.

Contexte de création Pour lui, le site fonctionne comme un outil de rédaction : il se connecte, crée un article, écrit son texte, sauvegarde, puis attend la publication.

Exemple Il prépare un article en brouillon, laisse draft: true, puis le passe à false seulement quand le texte est relu.

La personne qui demande des changements à Codex

Explication générale Si quelqu'un doit demander à Codex de modifier le site, cette personne doit pouvoir accéder au projet GitHub. Codex a besoin de lire les fichiers du site et, si on lui demande, de proposer ou pousser des changements.

Contexte de création Dans ce cas, il faut un compte GitHub et un accès ChatGPT/Codex. Netlify n'est nécessaire que si cette personne doit aussi gérer le déploiement ou les invitations.

Exemple Elle peut demander à Codex : "ajoute un paragraphe sur la page d'accueil, vérifie avec npm run build, puis pousse sur GitHub".

Le lecteur ou l'abonné newsletter

Explication générale Un lecteur n'a aucun compte à créer. Il lit le site public. S'il s'inscrit à la newsletter, il donne seulement son email dans le formulaire.

Contexte de création Le site public doit rester simple : les visiteurs n'ont pas besoin d'un accès interne pour lire.

Exemple Une personne lit un article sur chinetech.netlify.app et peut laisser son email sans créer de compte.

Créer un compte GitHub

  1. Aller sur github.com.
  2. Cliquer sur Sign up.
  3. Choisir un email fiable.
  4. Choisir un nom d'utilisateur simple.
  5. Valider l'email reçu.
  6. Activer la double authentification si GitHub le demande.

Créer l'accès CMS d'un rédacteur

  1. Dans Netlify, aller sur le site CHINETECH.
  2. Ouvrir Identity.
  3. Cliquer sur Invite users.
  4. Entrer l'email du rédacteur.
  5. Le rédacteur clique sur le lien reçu et choisit son mot de passe.
  6. Il se connecte ensuite sur /admin/.
Important : un lien d'invitation ou de récupération est secret. Ne jamais le coller dans ChatGPT, Slack, WhatsApp ou un document partagé.

Git et GitHub expliqués

Git est le principe de suivi des changements. GitHub est le site qui héberge ce suivi.

Git

Explication générale Git est comme un carnet de bord. À chaque changement important, on prend une photo de l'état du projet. Cette photo s'appelle un commit.

Contexte de création Git existe parce qu'un projet change souvent, et qu'il faut garder un historique fiable sans se perdre dans des copies de fichiers.

Exemple Après une modification du guide, on crée un commit pour garder une trace claire de ce changement.

GitHub

Explication générale GitHub est le bâtiment où l'on range ce carnet de bord. Il permet aussi à plusieurs personnes ou assistants de travailler sans perdre l'historique.

Contexte de création Git tout seul vit sur un ordinateur. GitHub ajoute un lieu partagé en ligne, pratique pour collaborer, relire, publier et connecter Netlify.

Exemple Le repo Duflosth/chinetech-blog est la version officielle du projet CHINETECH.

Les mots à connaître

Repository

Explication générale Le repository, souvent appelé repo, est le dossier du projet sur GitHub. Pour CHINETECH, c'est l'endroit où sont rangés le site, les articles, la configuration du CMS et les documents comme ce guide.

Contexte de création Il faut un endroit unique qui contient tout ce qui compose le projet, sinon les fichiers se dispersent.

Exemple Quand Codex travaille sur CHINETECH, il doit ouvrir le repo du blog, pas un autre dossier.

Commit

Explication générale Un commit est une sauvegarde nommée. Quand on change quelque chose et qu'on veut garder une trace propre, on crée un commit avec un message court, par exemple : "Simplifier le guide d'utilisation".

Contexte de création Sans commit, on ne sait pas précisément quand une modification a été faite ni pourquoi.

Exemple Après avoir changé cette documentation, on crée un commit du type "Clarify service explanations".

Branch

Explication générale Une branch est une copie de travail. Elle permet de tester une modification sans toucher directement à la version principale du site. Pour ce projet, la version publiée part de la branche master.

Contexte de création Quand plusieurs idées sont testées en même temps, il faut pouvoir les isoler pour ne pas casser la version publique.

Exemple Pour une grosse refonte visuelle, on peut travailler sur une branche séparée avant de publier.

Push

Explication générale Push veut dire envoyer les changements vers GitHub. Une fois le push fait, Netlify voit le changement et peut publier automatiquement la nouvelle version du site.

Contexte de création Un changement fait sur un ordinateur ou par Codex doit rejoindre GitHub pour devenir officiel.

Exemple Quand Codex pousse une modification sur master, Netlify démarre un nouveau déploiement.

Pull request

Explication générale Une pull request est une demande de relecture avant d'intégrer un changement. C'est utile quand plusieurs personnes travaillent ensemble et qu'on veut vérifier avant de publier.

Contexte de création Sur un projet à plusieurs, il faut parfois relire avant de fusionner une modification dans la version principale.

Exemple Si une personne propose une nouvelle page, on peut ouvrir une pull request pour la relire avant publication.

Markdown

Explication générale Markdown est une façon simple d'écrire du texte. On peut faire des titres, des listes, des liens et des paragraphes sans utiliser un logiciel compliqué. Les articles du blog sont stockés dans ce format.

Contexte de création Markdown existe pour écrire du contenu lisible à la fois par un humain et par un site web.

Exemple Dans un article, écrire ## Conclusion crée un titre de section.

Pour un non-technicien : il n'est pas nécessaire de manipuler Git. Le CMS et Codex peuvent faire le travail. Mais comprendre les mots aide à ne pas paniquer.

Écrire des articles

Le bon chemin pour écrire est le CMS : chinetech.netlify.app/admin/.

Pourquoi écrire dans le CMS ?

Explication générale Le CMS est l'outil normal pour écrire un article. Il présente un formulaire clair, puis transforme le contenu en fichier utilisable par le site.

Contexte de création Sans CMS, il faudrait ouvrir les fichiers du projet et écrire directement en Markdown. C'est possible, mais ce n'est pas confortable pour une personne non technique.

Exemple Pour publier un article sur DeepSeek, le rédacteur ouvre /admin/, crée un nouvel article, écrit son texte, puis passe draft à false.

Créer un article

  1. Se connecter à /admin/.
  2. Ouvrir Articles.
  3. Cliquer sur New Article.
  4. Remplir le titre, la description, la date, l'auteur et les tags.
  5. Écrire le texte dans le champ principal.
  6. Laisser draft: true tant que l'article n'est pas prêt.
  7. Quand tout est relu, passer draft: false.
  8. Sauvegarder.
  9. Attendre que Netlify publie le site.

Les champs les plus importants

Title

Explication générale title est le titre visible de l'article. Il doit être clair avant d'être brillant. Un bon titre permet de comprendre le sujet sans lire le reste.

Contexte de création Un site a besoin d'un titre pour afficher l'article dans les listes, les pages et les aperçus.

Exemple "DeepSeek change-t-il vraiment le coût de l'IA ?" est plus utile que "Révolution IA".

Description

Explication générale description est le résumé court. C'est la phrase qui aide le lecteur à décider s'il veut ouvrir l'article. Elle doit expliquer l'intérêt du sujet, pas répéter exactement le titre.

Contexte de création La description sert dans les cartes d'article et peut aider au partage du lien.

Exemple "Le modèle chinois interroge la façon dont les laboratoires réduisent leurs coûts d'entraînement."

Tags

Explication générale tags sert à classer l'article. Par exemple : IA, Business, Société, Puces, Mobilité. Il vaut mieux quelques tags cohérents qu'une longue liste vague.

Contexte de création Les tags aident à retrouver les sujets et à comprendre rapidement la famille d'un article.

Exemple Pour un article sur les semi-conducteurs chinois : IA, Puces, Industrie.

Draft

Explication générale draft contrôle la visibilité. Si draft vaut true, l'article reste invisible sur le site public. Quand l'article est prêt, on passe à false.

Contexte de création Un rédacteur doit pouvoir préparer un texte sans le publier immédiatement.

Exemple Pendant la relecture, garder draft: true. Après validation, mettre draft: false.

Featured

Explication générale featured permet de mettre un article en avant sur la page d'accueil. À utiliser seulement pour les articles vraiment importants, sinon la section "À la une" perd son sens.

Contexte de création La page d'accueil ne peut pas tout montrer. Le champ featured sert à choisir les articles prioritaires.

Exemple Un article de fond sur la stratégie chinoise de l'IA peut être mis en avant. Une brève correction ne devrait pas l'être.

Mini-recette d'un bon article

  • Un titre clair.
  • Une première phrase qui dit pourquoi le sujet compte.
  • Trois parties maximum si possible.
  • Des faits vérifiés.
  • Une conclusion simple : ce qu'il faut surveiller ensuite.

Modifier le site avec Codex

Codex sert à modifier le code du site : design, pages, formulaires, corrections, bugs. Il ne remplace pas le CMS pour écrire un article simple.

Deux façons d'utiliser Codex

Plus simple

Codex dans ChatGPT

Explication générale On connecte GitHub, on choisit le projet, puis on demande une modification. C'est le plus adapté pour une personne peu technique.

Contexte de création L'objectif est de pouvoir demander un changement en langage naturel, sans ouvrir soi-même les fichiers. Codex lit le projet, modifie ce qui est nécessaire et peut expliquer ce qu'il a changé.

Exemple "Dans le repo CHINETECH, modifie le texte de la newsletter, vérifie avec npm run build, puis pousse si tout passe."

Plus avancé

Codex dans le terminal

Explication générale On travaille depuis l'ordinateur, dans le dossier du projet. C'est utile pour quelqu'un qui sait déjà ouvrir un terminal.

Contexte de création Le terminal donne plus de contrôle : on peut lancer les commandes, voir les fichiers, tester localement et pousser les changements soi-même.

Exemple Une personne technique ouvre le dossier du projet, lance Codex, demande une modification, puis vérifie le résultat avec le navigateur et npm run build.

Quand utiliser Codex ?

  • Changer un texte présent dans la page d'accueil.
  • Ajouter une page statique.
  • Modifier le design.
  • Corriger un problème de lien ou d'affichage.
  • Ajouter un champ au CMS.
  • Générer un document HTML/PDF comme celui-ci.

Quand ne pas utiliser Codex ?

  • Pour écrire un article normal : utiliser le CMS.
  • Pour réinitialiser un mot de passe : utiliser Netlify Identity.
  • Pour envoyer un secret ou token : ne jamais le coller à un assistant.

Utiliser un LLM pour écrire

ChatGPT ou un autre LLM peut aider à préparer un article, mais il ne doit pas inventer les faits.

À quoi sert un LLM ici ?

Explication générale Un LLM sert à réfléchir, structurer, reformuler et préparer un brouillon. Il peut aider à rendre un article plus clair, mais il ne remplace pas la vérification humaine.

Contexte de création Les outils de rédaction IA sont utiles parce qu'ils accélèrent les tâches répétitives : proposer un plan, trouver un angle, améliorer une phrase, repérer les points flous.

Exemple Pour un article sur les taxis volants à Guangzhou, on peut demander un plan en trois parties, puis vérifier séparément les chiffres, dates et entreprises citées.

Bon usage

  • Demander un plan.
  • Demander une reformulation plus claire.
  • Demander une description courte.
  • Demander des titres alternatifs.
  • Demander les points à vérifier.

Routine simple

Pour éviter les erreurs, il suffit de suivre quelques habitudes simples.

Pour un nouvel éditeur

  1. L'inviter dans Netlify Identity.
  2. Lui faire créer son mot de passe.
  3. Lui envoyer le lien /admin/.
  4. Lui demander de créer un article test en brouillon.
  5. Vérifier ensemble que le champ draft est compris.

Avant publication

  • Le titre est clair.
  • Le résumé est court.
  • Les faits importants sont vérifiés.
  • Les tags sont cohérents.
  • draft est passé à false seulement à la fin.
  • Le site est vérifié après le déploiement.

Problèmes fréquents

Le lien d'invitation ne marche pas

Un lien d'invitation peut expirer ou être consommé après utilisation. La solution la plus simple est de renvoyer une nouvelle invitation depuis Netlify Identity. Il faut ensuite utiliser le lien le plus récent.

Rate limit exceeded

Ce message veut dire que trop de demandes ont été faites en peu de temps, par exemple trop d'emails de récupération de mot de passe. Il faut attendre 30 à 60 minutes, puis refaire une seule demande.

L'article n'apparaît pas

La première chose à vérifier est draft. Si draft vaut true, l'article est volontairement caché. Si draft vaut false, il faut attendre que Netlify finisse de publier le site.

Le site affiche 404

Vérifier d'abord l'adresse exacte : chinetech.netlify.app. Une lettre manquante suffit à tomber sur une page introuvable. Si l'adresse est bonne, regarder ensuite le dernier déploiement Netlify.

Le CMS ne publie pas

Si le CMS s'ouvre mais n'arrive pas à sauvegarder, il faut vérifier Netlify Identity et Git Gateway. En langage simple : l'utilisateur doit avoir le droit d'entrer, et le coursier doit pouvoir envoyer l'article vers GitHub.
