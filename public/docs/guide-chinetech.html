Vue d'ensemble

CHINETECH est un site statique : les pages sont fabriquées à l'avance, puis servies très rapidement. Cette approche est simple, peu coûteuse, robuste, et adaptée à un blog éditorial.

Écrire

Decap CMS

Interface visuelle pour créer les articles, choisir les tags, ajouter une image, garder un brouillon ou publier.

Stocker

GitHub

Bibliothèque officielle du projet. Chaque article et chaque fichier du site y sont versionnés.

Afficher

Netlify

Héberge le site, reconstruit les pages après une modification, gère les formulaires et les accès éditeurs.

Le parcours d'un article

  1. L'éditeur se connecte à /admin/.
  2. Il crée ou modifie un article dans Decap CMS.
  3. Decap enregistre le changement dans GitHub via Git Gateway.
  4. Netlify détecte le changement, lance un build, puis publie le nouveau site.
  5. L'article apparaît sur le site si draft vaut false.
À retenir : le CMS ne modifie pas directement le site visible. Il modifie les fichiers dans GitHub. Netlify reconstruit ensuite le site. Il peut donc y avoir un délai de 1 à 3 minutes.

Rôles et accès

Il y a plusieurs types de comptes. C'est souvent la partie la plus confuse, donc il faut bien distinguer les rôles.

Rôle Compte nécessaire Ce que la personne peut faire Conseil
Propriétaire du projet GitHub + Netlify dashboard Configurer le site, gérer les builds, inviter les éditeurs, voir les formulaires. À garder limité à une ou deux personnes fiables.
Éditeur non technique Netlify Identity seulement Se connecter au CMS, créer et modifier des articles. Le meilleur choix pour quelqu'un qui ne code pas.
Contributeur GitHub Compte GitHub Modifier les fichiers du projet, proposer des changements, accéder à l'historique. Utile si la personne comprend au moins les bases de GitHub.
Visiteur newsletter Aucun compte Lire le site et laisser son email dans le formulaire. Ses données arrivent dans Netlify Forms.

La règle simple

À faire

  • Pour écrire des articles : inviter dans Netlify Identity.
  • Pour relire ou conseiller : partager le site public ou un article en brouillon exporté.
  • Pour toucher au code : passer par GitHub et idéalement par une personne technique.

À éviter

  • Donner ton mot de passe Netlify ou GitHub.
  • Ajouter un non-technicien comme propriétaire Netlify sans besoin clair.
  • Partager un lien d'invitation ou de récupération dans un chat public.

Ajouter une personne non technique

Pour une personne qui doit seulement écrire ou corriger des articles, le plus propre est de l'ajouter comme utilisateur Netlify Identity, pas comme membre Netlify et pas forcément comme collaborateur GitHub.

Procédure recommandée

  1. Ouvrir Netlify : app.netlify.com.
  2. Aller sur le site CHINETECH.
  3. Ouvrir Identity.
  4. Vérifier que les inscriptions sont en mode Invite only.
  5. Cliquer sur Invite users.
  6. Saisir l'email de la personne.
  7. La personne reçoit un email d'invitation.
  8. Elle clique sur Accept the invite.
  9. Elle choisit son propre mot de passe.
  10. Elle va ensuite sur chinetech.netlify.app/admin/.
Important : si l'invitation ne marche pas, renvoyer une nouvelle invitation. Les liens contiennent un token temporaire. Un ancien lien peut expirer, être consommé, ou être bloqué si trop de demandes ont été faites.

Test d'onboarding à faire ensemble

  • La personne peut ouvrir /admin/.
  • Elle peut se connecter avec son email et son mot de passe.
  • Elle voit la collection Articles.
  • Elle crée un article test avec draft: true.
  • Elle sait sauvegarder sans publier.
  • Elle sait mettre draft: false quand l'article est prêt.

GitHub sans jargon

GitHub conserve les fichiers du site. Même si une personne écrit via le CMS, le résultat finit dans GitHub.

Créer un compte GitHub

  1. Aller sur github.com.
  2. Cliquer sur Sign up.
  3. Choisir un email fiable, idéalement le même que pour les projets professionnels.
  4. Choisir un nom d'utilisateur lisible.
  5. Vérifier son email.
  6. Activer la double authentification si GitHub le demande ou la propose.

Faut-il créer un compte GitHub pour l'éditeur non technique ?

Pas nécessaire pour publier via Decap CMS. Le compte utile pour lui est Netlify Identity. Le compte GitHub devient utile si la personne doit relire des fichiers, ouvrir des tickets, ou intervenir directement sur le dépôt.

Inviter quelqu'un sur le repo GitHub

  1. Ouvrir github.com/Duflosth/chinetech-blog.
  2. Cliquer sur Settings.
  3. Aller dans Collaborators ou Collaborators & teams.
  4. Cliquer sur Add people.
  5. Rechercher son nom d'utilisateur GitHub.
  6. Choisir le rôle le plus faible qui suffit.
Conseil concret : pour une personne non technique, ne commence pas par GitHub. Commence par le CMS. GitHub est le filet de sécurité et l'historique, pas l'outil d'écriture quotidien.

Netlify, déploiement et comptes

Netlify héberge le site public. Quand GitHub change, Netlify reconstruit le site et publie une nouvelle version.

Ce qui doit rester configuré

  • Site Netlify connecté au repo Duflosth/chinetech-blog.
  • Branche de production : master.
  • Build command : npm run build.
  • Publish directory : dist.
  • Identity activé.
  • Registration preferences : Invite only.
  • Git Gateway activé.
  • Forms detection activé pour la newsletter.

Les deux types de comptes Netlify

Type À quoi ça sert Pour qui
Compte Netlify dashboard Gérer le site, les builds, les domaines, Identity, Forms, Git Gateway. Propriétaire du projet.
Compte Netlify Identity Se connecter au CMS du site. Éditeur non technique.

Un éditeur n'a pas besoin de compte dashboard Netlify. Il a besoin d'un compte Identity du site, créé via une invitation.

Après une modification, combien de temps attendre ?

En général, Netlify prend une à quelques minutes pour reconstruire le site. Si l'article n'apparaît pas immédiatement, vérifier d'abord le champ draft, puis l'onglet Deploys dans Netlify.

Decap CMS et création d'articles

Le CMS est l'interface la plus importante pour une personne non technique. Son adresse est https://chinetech.netlify.app/admin/.

Créer un article pas à pas

  1. Ouvrir /admin/.
  2. Se connecter avec l'email invité et le mot de passe choisi.
  3. Cliquer sur Articles.
  4. Cliquer sur New Article ou équivalent.
  5. Remplir le titre.
  6. Rédiger une description courte.
  7. Choisir la date.
  8. Mettre l'auteur.
  9. Ajouter les tags.
  10. Ajouter une image de couverture si disponible.
  11. Garder draft: true tant que ce n'est pas prêt.
  12. Écrire le contenu dans le champ Markdown.
  13. Sauvegarder.
  14. Quand tout est relu, passer draft à false.
  15. Attendre le déploiement Netlify.

À quoi servent les champs ?

Champ Usage Conseil
title Titre visible dans les cartes et la page article. Clair, concret, pas trop long.
description Résumé court de l'article. Une ou deux phrases. Dire pourquoi l'article mérite d'être lu.
date Date de publication. La page articles trie du plus récent au plus ancien.
author Nom de l'auteur. Garder un format stable : "Rédaction CHINETECH" ou prénom + nom.
tags Catégories éditoriales. Limiter à 2 ou 3 tags utiles.
cover Image de couverture. Utiliser une image horizontale, claire, sans texte minuscule.
draft Masque ou publie l'article. true = brouillon invisible. false = publié.
featured Met l'article dans la section "À la une". Réserver à 1 à 4 articles importants.
featuredRank Ordre des articles à la une. 1 est prioritaire, puis 2, etc.

Méthode éditoriale CHINETECH

Le blog doit donner une impression d'analyse, pas de bruit d'actualité. Chaque article doit aider le lecteur à comprendre une dynamique.

Structure conseillée

  1. Angle : quelle question l'article résout-il ?
  2. Contexte : pourquoi le sujet compte maintenant ?
  3. Analyse : quels mécanismes économiques, technologiques ou politiques sont à l'oeuvre ?
  4. Conséquences : qui gagne, qui perd, qui doit surveiller quoi ?
  5. À suivre : 3 signaux concrets à observer.

Checklist avant publication

  • Le titre est compréhensible sans contexte.
  • La description donne envie sans survendre.
  • Le premier paragraphe explique pourquoi le sujet compte.
  • Chaque affirmation factuelle importante est vérifiée.
  • Les dates sont exactes.
  • Les noms d'entreprises sont bien orthographiés.
  • Les acronymes sont expliqués à la première mention.
  • L'article ne contient pas de promesse vague comme "révolutionnaire" sans preuve.
  • Le champ draft reste true jusqu'à validation.
  • Après publication, l'article est vérifié sur mobile et desktop.

Conseils de style

Bon ton

  • Précis, calme, analytique.
  • Expliquer les enjeux sans jargon inutile.
  • Préférer les mécanismes aux superlatifs.
  • Dire ce qu'on sait, ce qu'on ne sait pas, et ce qu'il faut surveiller.

À éviter

  • Copier-coller un communiqué.
  • Écrire uniquement à partir d'un buzz LinkedIn/X.
  • Publier une info récente sans date ni source.
  • Utiliser une image sans droit ou sans origine claire.

Tags recommandés

Utiliser des tags stables permet de garder une ligne éditoriale claire : Tech, IA, Innovation, Business, Société, Mobilité, Semi-conducteurs, Géopolitique, Startups.

Que demander à un LLM

Un LLM est utile pour structurer, reformuler, trouver des angles et vérifier la clarté. Il ne doit pas être utilisé comme source unique de vérité.

Règles d'or

  • Ne jamais donner de mot de passe, token, lien d'invitation ou lien de récupération.
  • Demander au LLM de signaler les incertitudes.
  • Vérifier séparément les faits récents.
  • Demander une version Markdown prête à coller dans le CMS.
  • Garder la décision éditoriale humaine.

Comment demander une modification de site

Une bonne demande contient toujours :

  • la page concernée ;
  • ce qui ne va pas ;
  • le résultat souhaité ;
  • une capture d'écran si possible ;
  • la contrainte à ne pas casser.

Exemple clair

Sur la homepage de CHINETECH, dans la section newsletter,
je veux remplacer le texte "chaque vendredi matin" par "deux fois par mois".

Ne change pas le formulaire Netlify.
Vérifie que npm run build passe.
Pousse sur master si tout est bon.

Newsletter et formulaires

Le formulaire newsletter utilise Netlify Forms. Cela évite de mettre un token d'emailing dans le navigateur.

Où voir les inscriptions ?

  1. Ouvrir Netlify dashboard.
  2. Aller sur le site CHINETECH.
  3. Ouvrir Forms.
  4. Choisir le formulaire newsletter.
  5. Exporter ou traiter les emails.

Que faire ensuite avec les emails ?

  • Exporter régulièrement en CSV si la liste est petite.
  • Connecter plus tard à Brevo, Resend, Mailchimp, Zapier ou Make si besoin.
  • Informer clairement les inscrits de la fréquence d'envoi.
  • Prévoir un moyen de désinscription avant d'envoyer une vraie newsletter commerciale.
Conseil concret : tant que la newsletter n'est pas automatisée, noter la date de chaque export et éviter les doublons.

Sécurité et bonnes pratiques

Le site est simple, mais les accès restent importants. Les erreurs les plus graves sont souvent humaines : partage de liens secrets, mots de passe faibles, mauvais rôle donné à la mauvaise personne.

À ne jamais partager

  • Mot de passe GitHub.
  • Mot de passe Netlify.
  • Lien contenant #invite_token=.
  • Lien contenant #recovery_token=.
  • Token API GitHub, Netlify, Resend, Brevo ou autre.
  • Capture d'écran montrant des clés secrètes.

Bonnes habitudes

  • Utiliser un gestionnaire de mots de passe.
  • Activer la double authentification sur GitHub.
  • Garder Netlify en mode Invite only.
  • Retirer l'accès d'une personne qui ne collabore plus.
  • Relire les articles avant de passer draft à false.
  • Ne pas installer de dépendances ou services payants sans décision explicite.

Gestion des erreurs

Si un article publié contient une erreur grave, la réponse la plus simple est de retourner dans le CMS, remettre draft: true, sauvegarder, attendre le redeploy, puis corriger calmement.

Problèmes fréquents

Le lien d'invitation renvoie juste à l'accueil

  1. Vérifier que l'URL est bien https://chinetech.netlify.app/.
  2. Ne pas utiliser l'ancien domaine legendary-sundae-f81d8a.netlify.app.
  3. Ouvrir le lien dans une fenêtre privée.
  4. Si le lien a déjà été utilisé, renvoyer une nouvelle invitation.
  5. Si trop d'essais ont été faits, attendre avant de renvoyer un email.

"Rate limit exceeded"

Cela signifie que trop d'emails d'invitation ou de récupération ont été demandés en peu de temps. Attendre 30 à 60 minutes, parfois plus, puis refaire une seule demande.

Le CMS ne se connecte pas à GitHub

  • Vérifier que Git Gateway est activé dans Netlify.
  • Vérifier que Identity est activé.
  • Vérifier que l'utilisateur est bien invité.
  • Vérifier que /admin/config.yml répond bien dans le navigateur.

L'article n'apparaît pas sur le site

  • Vérifier draft : si c'est true, c'est normal qu'il soit invisible.
  • Vérifier la date et le titre.
  • Attendre le redeploy Netlify.
  • Dans Netlify, ouvrir Deploys et regarder si le build a échoué.

Le formulaire newsletter ne remonte pas

  • Vérifier Netlify Forms.
  • Vérifier que la détection des formulaires est active.
  • Tester avec une adresse email non jetable.
  • Vérifier le dossier spam si une notification email est configurée.

Le site affiche 404

  • Vérifier l'orthographe : chinetech.netlify.app, pas chintech.netlify.app.
  • Vérifier que le site n'a pas changé de nom.
  • Vérifier dans Netlify que le dernier deploy est réussi.
  • Vérifier que le dossier de publication est bien dist.

Checklists prêtes à l'emploi

Avant d'ajouter un éditeur

  • Le rôle exact de la personne est clair.
  • Elle a besoin du CMS, pas forcément de GitHub.
  • Identity est en mode Invite only.
  • Git Gateway est activé.
  • Le lien /admin/ fonctionne.
  • Un court appel d'onboarding est prévu.

Avant de publier un article

  • Titre relu.
  • Description relue.
  • Corps relu.
  • Faits importants vérifiés.
  • Image utilisable légalement.
  • Tags cohérents.
  • draft passé à false seulement à la fin.
  • Article vérifié après publication.

Avant de demander une modification à un assistant IA

  • Dire quelle page ou quel fichier est concerné.
  • Dire le résultat attendu.
  • Dire ce qu'il ne faut pas casser.
  • Demander de lancer npm run build.
  • Demander un résumé des fichiers modifiés.
  • Ne jamais coller de secret ou de token.

Routine mensuelle

  • Vérifier les utilisateurs Identity actifs.
  • Retirer les accès inutiles.
  • Exporter les emails newsletter si nécessaire.
  • Vérifier que le dernier deploy Netlify réussit.
  • Regarder les articles à la une et enlever ceux qui sont périmés.
  • Sauvegarder les décisions éditoriales importantes dans un document partagé.
