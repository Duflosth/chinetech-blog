Gérer le blog CHINETECH

Ce guide explique les services, les comptes à créer, comment écrire des articles, et comment demander à Codex de modifier le site sans rentrer dans la technique.

Les services

Voici les services à connaître. Il ne faut pas les apprendre comme une liste technique. Il faut plutôt les voir comme les personnes qui font tourner un petit magazine.

GitHub : l'armoire officielle

GitHub est l'endroit où le projet vit réellement. Il garde les fichiers du site, les articles, les images importantes et l'historique des changements. Dans notre métaphore, c'est l'armoire officielle du magazine : si quelque chose est dans GitHub, le projet peut le retrouver.

C'est aussi une machine à remonter le temps. Quand quelqu'un modifie le site, GitHub garde une trace. On peut donc comprendre qui a changé quoi, quand, et pourquoi.

Netlify : l'imprimeur et la vitrine

Netlify prend ce qui est rangé dans GitHub et le transforme en site visible par le public. Dès qu'un changement est envoyé dans GitHub, Netlify reconstruit le site et le met en ligne.

La différence est simple : GitHub garde les fichiers, Netlify montre le résultat. GitHub est la réserve du magasin. Netlify est la vitrine que les visiteurs voient.

Decap CMS : le bureau de rédaction

Decap CMS est la page d'administration accessible sur /admin/. C'est là qu'une personne non technique peut écrire un article avec un formulaire. Elle n'a pas besoin de connaître le code, ni d'ouvrir GitHub.

Quand l'article est sauvegardé, Decap CMS prépare le changement et l'envoie vers GitHub. Ensuite Netlify publie la nouvelle version du site.

Netlify Identity : le badge d'entrée

Netlify Identity sert à savoir qui a le droit d'entrer dans le CMS. Une personne invitée reçoit un email, crée son mot de passe, puis peut se connecter à /admin/.

Important : il ne faut pas créer un compte Netlify complet pour chaque rédacteur. Pour écrire des articles, une invitation Identity suffit.

Git Gateway : le coursier discret

Git Gateway est le lien entre le CMS et GitHub. Quand un rédacteur clique sur sauvegarder, Git Gateway transporte le changement vers l'armoire GitHub.

Normalement, un rédacteur ne voit presque jamais Git Gateway. Il doit seulement savoir que si le CMS n'arrive plus à publier, Git Gateway fait partie des éléments à vérifier dans Netlify.

Astro : la machine de mise en page

Astro est l'outil qui transforme les fichiers du projet en pages web rapides. Il prend les articles, les styles, les pages et les assemble pour produire le site final.

Une personne non technique n'a pas besoin d'utiliser Astro directement. Il travaille en arrière-plan pendant le déploiement Netlify.

Codex : l'assistant pour modifier l'atelier

Codex sert quand il faut changer le site lui-même : modifier la page d'accueil, corriger un bug, changer un style, ajouter une page, ajuster le CMS ou produire un document comme celui-ci.

Pour un article normal, on utilise le CMS. Pour changer le fonctionnement ou l'apparence du site, on demande à Codex de modifier le code dans GitHub.

Résumé très court : GitHub garde, Netlify publie, Decap CMS permet d'écrire, Identity ouvre la porte, Git Gateway transporte, Astro assemble, Codex modifie le site quand on lui demande.

Quels comptes créer ?

Il ne faut pas créer tous les comptes pour tout le monde. Chaque personne doit avoir seulement les accès qui correspondent à son rôle.

Le propriétaire du projet

Le propriétaire doit avoir un compte GitHub, un compte Netlify et un accès ChatGPT/Codex. C'est la personne qui peut gérer le site, inviter quelqu'un, vérifier les déploiements et demander des changements plus profonds.

C'est l'équivalent du directeur de publication : il n'écrit pas forcément tous les articles, mais il garde les clés principales.

Le rédacteur non technique

Un rédacteur qui écrit seulement des articles n'a pas besoin d'un compte GitHub. Il a besoin d'une invitation Netlify Identity pour entrer dans le CMS sur /admin/.

Pour lui, le site fonctionne comme un outil de rédaction : il se connecte, crée un article, écrit son texte, sauvegarde, puis attend la publication.

La personne qui demande des changements à Codex

Si quelqu'un doit demander à Codex de modifier le site, cette personne doit pouvoir accéder au projet GitHub. Codex a besoin de lire les fichiers du site et, si on lui demande, de proposer ou pousser des changements.

Dans ce cas, il faut un compte GitHub et un accès ChatGPT/Codex. Netlify n'est nécessaire que si cette personne doit aussi gérer le déploiement ou les invitations.

Le lecteur ou l'abonné newsletter

Un lecteur n'a aucun compte à créer. Il lit le site public. S'il s'inscrit à la newsletter, il donne seulement son email dans le formulaire.

Créer un compte GitHub

  1. Aller sur github.com.
  2. Cliquer sur Sign up.
  3. Choisir un email fiable.
  4. Choisir un nom d'utilisateur simple.
  5. Valider l'email reçu.
  6. Activer la double authentification si GitHub le demande.

Créer l'accès CMS d'un rédacteur

  1. Dans Netlify, aller sur le site CHINETECH.
  2. Ouvrir Identity.
  3. Cliquer sur Invite users.
  4. Entrer l'email du rédacteur.
  5. Le rédacteur clique sur le lien reçu et choisit son mot de passe.
  6. Il se connecte ensuite sur /admin/.
Important : un lien d'invitation ou de récupération est secret. Ne jamais le coller dans ChatGPT, Slack, WhatsApp ou un document partagé.

Git et GitHub expliqués

Git est le principe de suivi des changements. GitHub est le site qui héberge ce suivi.

Git

Git est comme un carnet de bord. À chaque changement important, on prend une photo de l'état du projet. Cette photo s'appelle un commit.

GitHub

GitHub est le bâtiment où l'on range ce carnet de bord. Il permet aussi à plusieurs personnes ou assistants de travailler sans perdre l'historique.

Les mots à connaître

Repository

Le repository, souvent appelé repo, est le dossier du projet sur GitHub. Pour CHINETECH, c'est l'endroit où sont rangés le site, les articles, la configuration du CMS et les documents comme ce guide.

Commit

Un commit est une sauvegarde nommée. Quand on change quelque chose et qu'on veut garder une trace propre, on crée un commit avec un message court, par exemple : "Simplifier le guide d'utilisation".

Branch

Une branch est une copie de travail. Elle permet de tester une modification sans toucher directement à la version principale du site. Pour ce projet, la version publiée part de la branche master.

Push

Push veut dire envoyer les changements vers GitHub. Une fois le push fait, Netlify voit le changement et peut publier automatiquement la nouvelle version du site.

Pull request

Une pull request est une demande de relecture avant d'intégrer un changement. C'est utile quand plusieurs personnes travaillent ensemble et qu'on veut vérifier avant de publier.

Markdown

Markdown est une façon simple d'écrire du texte. On peut faire des titres, des listes, des liens et des paragraphes sans utiliser un logiciel compliqué. Les articles du blog sont stockés dans ce format.

Pour un non-technicien : il n'est pas nécessaire de manipuler Git. Le CMS et Codex peuvent faire le travail. Mais comprendre les mots aide à ne pas paniquer.

Écrire des articles

Le bon chemin pour écrire est le CMS : chinetech.netlify.app/admin/.

Créer un article

  1. Se connecter à /admin/.
  2. Ouvrir Articles.
  3. Cliquer sur New Article.
  4. Remplir le titre, la description, la date, l'auteur et les tags.
  5. Écrire le texte dans le champ principal.
  6. Laisser draft: true tant que l'article n'est pas prêt.
  7. Quand tout est relu, passer draft: false.
  8. Sauvegarder.
  9. Attendre que Netlify publie le site.

Les champs les plus importants

Title

title est le titre visible de l'article. Il doit être clair avant d'être brillant. Un bon titre permet de comprendre le sujet sans lire le reste.

Description

description est le résumé court. C'est la phrase qui aide le lecteur à décider s'il veut ouvrir l'article. Elle doit expliquer l'intérêt du sujet, pas répéter exactement le titre.

Tags

tags sert à classer l'article. Par exemple : IA, Business, Société, Puces, Mobilité. Il vaut mieux quelques tags cohérents qu'une longue liste vague.

Draft

draft contrôle la visibilité. Si draft vaut true, l'article reste invisible sur le site public. Quand l'article est prêt, on passe à false.

Featured

featured permet de mettre un article en avant sur la page d'accueil. À utiliser seulement pour les articles vraiment importants, sinon la section "À la une" perd son sens.

Mini-recette d'un bon article

  • Un titre clair.
  • Une première phrase qui dit pourquoi le sujet compte.
  • Trois parties maximum si possible.
  • Des faits vérifiés.
  • Une conclusion simple : ce qu'il faut surveiller ensuite.

Modifier le site avec Codex

Codex sert à modifier le code du site : design, pages, formulaires, corrections, bugs. Il ne remplace pas le CMS pour écrire un article simple.

Deux façons d'utiliser Codex

Plus simple

Codex dans ChatGPT

On connecte GitHub, on choisit le projet, puis on demande une modification. C'est le plus adapté pour une personne peu technique.

Plus avancé

Codex dans le terminal

On travaille depuis l'ordinateur, dans le dossier du projet. C'est utile pour quelqu'un qui sait déjà ouvrir un terminal.

Quand utiliser Codex ?

  • Changer un texte présent dans la page d'accueil.
  • Ajouter une page statique.
  • Modifier le design.
  • Corriger un problème de lien ou d'affichage.
  • Ajouter un champ au CMS.
  • Générer un document HTML/PDF comme celui-ci.

Quand ne pas utiliser Codex ?

  • Pour écrire un article normal : utiliser le CMS.
  • Pour réinitialiser un mot de passe : utiliser Netlify Identity.
  • Pour envoyer un secret ou token : ne jamais le coller à un assistant.

Routine simple

Pour éviter les erreurs, il suffit de suivre quelques habitudes simples.

Pour un nouvel éditeur

  1. L'inviter dans Netlify Identity.
  2. Lui faire créer son mot de passe.
  3. Lui envoyer le lien /admin/.
  4. Lui demander de créer un article test en brouillon.
  5. Vérifier ensemble que le champ draft est compris.

Avant publication

  • Le titre est clair.
  • Le résumé est court.
  • Les faits importants sont vérifiés.
  • Les tags sont cohérents.
  • draft est passé à false seulement à la fin.
  • Le site est vérifié après le déploiement.

Problèmes fréquents

Le lien d'invitation ne marche pas

Un lien d'invitation peut expirer ou être consommé après utilisation. La solution la plus simple est de renvoyer une nouvelle invitation depuis Netlify Identity. Il faut ensuite utiliser le lien le plus récent.

Rate limit exceeded

Ce message veut dire que trop de demandes ont été faites en peu de temps, par exemple trop d'emails de récupération de mot de passe. Il faut attendre 30 à 60 minutes, puis refaire une seule demande.

L'article n'apparaît pas

La première chose à vérifier est draft. Si draft vaut true, l'article est volontairement caché. Si draft vaut false, il faut attendre que Netlify finisse de publier le site.

Le site affiche 404

Vérifier d'abord l'adresse exacte : chinetech.netlify.app. Une lettre manquante suffit à tomber sur une page introuvable. Si l'adresse est bonne, regarder ensuite le dernier déploiement Netlify.

Le CMS ne publie pas

Si le CMS s'ouvre mais n'arrive pas à sauvegarder, il faut vérifier Netlify Identity et Git Gateway. En langage simple : l'utilisateur doit avoir le droit d'entrer, et le coursier doit pouvoir envoyer l'article vers GitHub.
